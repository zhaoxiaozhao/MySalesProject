# Salesforce Apex 代码风格

用于统一 Apex、Trigger 和控制器代码的排版、注释和方法组织方式。

## 代码格式

- 缩进必须一致。
- `if`、`for`、`while`、`do` 等语句尽量单独成行。
- 无论分支体是否只有一行，都要显式使用 `{}`。
- 左大括号单独换行，避免把控制语句和 `{` 挤在同一行。

推荐风格：

```apex
if (condition)
{
    doWork();
}
```

不推荐风格：

```apex
if (condition){
    doWork();
}
```

## 嵌套控制

- 尽量使用浅嵌套。
- 避免在循环内叠加多层条件。
- 优先用提前返回、`continue`、拆分方法降低层级。

推荐优先判断不满足条件的分支，再尽早 `continue`，而不是把主逻辑包进多层 `if`。

## 注释规范

### 组件头注释

类和 Trigger 顶部应包含：

- 作者
- 创建时间
- 功能说明
- 约束信息（可选）
- 修改人
- 修改时间
- 修改内容

推荐模板：

```apex
/*
作者: Michael
创建时间: 2013-7-11
功能: 核心功能说明
约束: 可选
修改人: Sunny
修改时间: 2013-7-12
修改内容: 修改摘要
*/
```

### 方法注释

几乎每个方法都应在声明前写说明，至少覆盖：

- 方法用途
- 参数说明
- 返回值说明
- 异常说明
- 特殊约束或副作用

推荐模板：

```apex
/*
描述: 单个发送短信方法
参数 Mobile: 手机号码
参数 Content: 发送内容
返回值: 发送结果
*/
public SMSResult sendSingleSMS(String mobile, String content)
{
}
```

### 变量与关键代码块

- 关键变量要有注释。
- 复杂查询、复杂分支、关键业务块要有说明。
- 不写空泛废话，只解释业务意图或约束原因。

## 运算符与可读性

- 二元运算符两侧必须加空格。
- 示例：`a = b + c`

## 行长与拆行

- 单行建议控制在 70 到 80 字符内。
- 长表达式应在低优先级运算符处换行。
- 换行后运算符放在新行行首，并保持整齐缩进。

## 方法长度与模块化

- 不推荐单个方法超过 100 行。
- 大代码块应拆到独立方法。
- 多个组件存在相似逻辑时，应提取公用方法。

## 输出要求

生成代码时，默认执行以下动作：

1. 采用统一的大括号与缩进风格。
2. 为类、Trigger、公共方法补齐说明性注释。
3. 主动拆分过长方法和深嵌套逻辑。
