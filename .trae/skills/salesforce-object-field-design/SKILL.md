# Salesforce 对象与字段设计

用于在 Salesforce 中设计业务数据模型，统一对象、字段、关系、约束和配置化方案。

## 适用场景

- 新增业务对象
- 在标准对象上扩展字段
- 设计自定义字段
- 设计 `Lookup` 或 `Master-Detail`
- 设计 `Record Type`
- 设计字段集、页面布局、校验规则
- 评审现有对象模型是否合理

## 设计顺序

1. 先判断使用标准对象还是新建自定义对象。
2. 明确对象的业务职责、主键语义和生命周期。
3. 设计字段清单，再设计字段类型和必填性。
4. 设计对象关系，再判断是 `Lookup` 还是 `Master-Detail`。
5. 设计展示与配置项，例如 `Field Set`、页面布局、`Record Type`。
6. 最后检查命名、可维护性和跨环境可迁移性。

## 对象设计原则

- 优先复用标准对象，只有标准对象无法承载业务语义时再新建自定义对象。
- 一个对象应只承载一类清晰的业务实体，不要把多个职责揉进同一个对象。
- 对象命名遵循 `salesforce-naming-standard`。
- 中文标签可以面向业务用户，API 名称必须使用有意义的英文。

## 字段设计原则

- 字段名必须直接表达业务含义，禁止拼音和无意义缩写。
- 字段类型按业务含义选，不按页面表现选。
- 能用配置表达的业务约束，不要写死在代码里。
- 只在确有业务必要时设置必填、唯一、默认值。
- 对可能频繁变化的展示字段，优先考虑 `Field Set`。

## 字段类型选择

- 文本标识、编码、名称：优先 `Text`
- 长说明、备注：优先 `Long Text Area`
- 金额、数量、比率：按精度选择 `Number` / `Currency` / `Percent`
- 日期类：按是否需要时间选择 `Date` / `DateTime`
- 固定候选值：优先 `Picklist`
- 多选但确有业务必要时才用 `Multi-Select Picklist`
- 计算结果优先 `Formula`
- 布尔开关优先 `Checkbox`

## 关系设计

### 选择 `Lookup`

适用于：

- 子记录可以脱离父记录独立存在
- 删除父记录时不希望自动删除子记录
- 关系较松，生命周期不绑定

### 选择 `Master-Detail`

适用于：

- 子记录完全依附父记录存在
- 删除父记录时应联动删除子记录
- 需要继承所有者、共享或汇总字段能力

## 约束设计

- 简单字段合法性优先用验证规则，不要先写 Trigger。
- 可推导值优先用公式字段，不要先写 Apex。
- 枚举值优先用 Picklist，不要先写文本字段再靠代码校验。
- 严禁通过硬编码 ID、`RecordTypeId`、用户名、角色名来完成对象配置。

## 配置化设计

- 展示字段变化频繁时，优先设计 `Field Set`。
- 页面差异明显时，再考虑 `Record Type + Page Layout`。
- 配置项能落在元数据或配置对象里，就不要放进 Apex 常量。

## 输出要求

当你执行对象/字段设计任务时，按下面结构输出：

1. 对象设计结论
2. 字段清单
3. 关系设计结论
4. 校验与配置化方案
5. 命名建议
6. 需要避免的实现风险

## 字段清单模板

```text
对象: Enrollment__c
- Student__c: Lookup(Student__c), 必填, 学员
- Course__c: Lookup(Course__c), 必填, 课程
- Status__c: Picklist, 必填, 状态
- Enroll_Date__c: Date, 必填, 报名日期
- Amount__c: Currency(16,2), 选填, 报名金额
```

## 自检清单

- 是否误建了本可复用的标准对象
- 是否存在语义混乱的对象职责
- 是否存在不合理字段类型
- 是否把 `Lookup` 和 `Master-Detail` 选反
- 是否本可用验证规则/公式字段却写成代码
- 是否需要 `Field Set`、页面布局或 `Record Type`
- 是否存在硬编码配置的风险
